## Solver: `_2x2/Solver2x2.py` (new, 290 lines)

Dedicated corner-only beginner method:
1. **L1 (bottom corners):** Bring white face up, solve 4 white corners into position+orientation
2. **L3 position:** Permute top corners using A-perm (`U R U' L' U R' U' L`)
   - Handles all S‚ÇÑ permutation types: identity, 4-cycle (U rotations), 3-cycle (A-perm),
     double transposition (A-perm ‚Üí 3-cycle), transposition (U ‚Üí 3-cycle)
3. **L3 orient:** Twist each top corner using `(R' D' R D) √ó 2`, rotate U between corners

Routing: `Solvers.default()` returns `Solver2x2` when `cube.size == 2`.
`Solvers.by_name()` redirects all non-TWO_BY_TWO solvers to `two_by_two()` for size 2.

## Domain Model Changes ‚Äî CONCERNS

The 2x2 model changes are **invasive and spread across many files**. The approach creates
"empty" Edge and Center objects (with no slices) and then guards every access with
`if not self._slices` checks scattered throughout the model. This is fragile.

### Center.py ‚Äî Virtual color tracking

- Added `_virtual_color: Color | None` field ‚Äî tracks face color through whole-cube rotations
- Added `_face_ref: Face | None` ‚Äî back-reference since empty centers can't derive face from slices
- `__init__` now accepts `face` parameter; sets `self._cube` directly for 2x2 (bypasses Part.__init__ logic)
- `face` property: `if not self._slices: return self._face_ref`
- `_3x3_representative_edges`: returns `()` for empty centers
- `is3x3`: returns `True` for empty centers (trivially reduced)
- `edg()`: raises `ValueError` for empty centers
- `color` property: returns `_virtual_color` instead of `edg().color` for 2x2
- `clone()`: copies `_face_ref` and `_virtual_color`
- `__str__`: special case for empty centers

**Concern:** The `_virtual_color` mechanism is a significant new concept that exists only for 2x2.
It adds state that must be correctly maintained during rotations (see Slice.py changes).

### Edge.py ‚Äî Empty slices handling

- `__init__`: sets `self._cube = f1.cube` before `super().__init__()` when slices empty;
  skips face assertions for empty edges
- `_3x3_representative_edges`: returns `()` for empty edges
- `is3x3`: returns `True` for empty edges
- `get_other_face_edge()`: raises `ValueError` for empty edges
- `get_other_face()`: uses `_f1`/`_f2` references directly for 2x2
- `single_shared_face()`: raises `ValueError` for empty edges
- `name` property: uses stored `_f1`/`_f2` face names instead of slice-based lookup

**Concern:** Many methods raise `ValueError` for 2x2 ‚Äî these are runtime bombs if any code
path accidentally calls them on a 2x2 edge.

### Part.py ‚Äî Guard every `_3x3_representative_edges` access

- `__init__`: `next(self.all_slices, None)` fallback; uses pre-set `self._cube` for empty parts
- `finish_init()`: sets `_fixed_id = frozenset()` for empty parts (no slices to derive ID from)
- `has_slices` property (new): `next(self.all_slices, None) is not None`
- `fixed_id` assertion: changed `assert self._fixed_id` ‚Üí `assert self._fixed_id is not None`
  (because `frozenset()` is falsy)
- `get_face_edge()`: raises `ValueError` for empty parts
- `__str__`: returns `"(PartName:empty)"` for empty parts
- `match_face()`: returns `True` for empty parts (trivially matched)
- `position_id`: returns `frozenset()` for empty parts
- `colors_id`: returns `frozenset()` for empty parts
- `find_face_by_color()`: raises `ValueError` for empty parts

**Concern:** ~10 methods in Part.py now have `if not rep_edges` guards. This is the core of the
fragility ‚Äî every new method added to Part must remember to handle empty parts.

### Slice.py ‚Äî Virtual color rotation

- `rotate()`: early return for `n_slices == 0`, delegates to `_rotate_virtual_center_colors()`
- New `_VIRTUAL_COLOR_CYCLES` dict: maps M/E/S slice names to face-name cycles
- New `_rotate_virtual_center_colors()`: cycles `_virtual_color` on centers for whole-cube rotations
- Accesses `self.cube._faces` directly (private attribute, `type: ignore`)

**Concern:** Accesses `cube._faces` private dict. The virtual color cycle is a parallel
rotation system that must stay in sync with physical rotations. If someone adds a new
rotation path that doesn't go through `Slice.rotate()`, virtual colors will desync.

### Face.py

- `_create_center()`: passes `face=self` to Center constructor
- `finish_init()`: early return for `n_slices == 0` (no markers to set)
- `__str__`: uses `self.color` instead of `self._center.edg().color` (works for 2x2)
- `is_solved`: special 2x2 path ‚Äî checks only corners (no edges/center)
- `is3x3`: returns `True` for `n_slices == 0`

### Cube.py

- `get_3x3_colors()`: wraps edge iteration in `if self.n_slices > 0`
- Centers dict: uses `face.color` instead of `face.center.get_slice((0,0)).edges[0].color`

### CubeSanity.py

- `_sanity_3x3()`: skips for `n_slices == 0`
- `_sanity_center_distribution()`: skips for `n_slices == 0`
- `_sanity_edge_distribution()`: skips for `n_slices == 0`

### Scramble.py

- Skip slice randomization when `max_slice < 1`

## Known Issues / Design Debt

1. **Scattered `if not self._slices` guards** ‚Äî fragile, every new Part/Edge/Center method
   must remember to handle the empty case. A better approach might be:
   - Null object pattern: dedicated `EmptyEdge`/`EmptyCenter` subclasses
   - Or: don't create Edge/Center objects at all for 2x2, restructure Face to be optional
2. **Virtual color system** ‚Äî parallel state (`_virtual_color`) that must stay synchronized
   with physical rotations via `_rotate_virtual_center_colors`. Accesses private `cube._faces`.
3. **`frozenset()` as fixed_id** ‚Äî empty frozenset is valid but breaks `__hash__` uniqueness
   assumptions if multiple empty parts exist in the same container.
4. **RuntimeError landmines** ‚Äî several methods raise `ValueError("...2x2 cube")` that would
   crash at runtime if any solver code accidentally calls them for 2x2.
